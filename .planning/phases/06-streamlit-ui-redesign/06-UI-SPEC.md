# Phase 06 — UI Design Contract

> Visual and interaction contract for the Streamlit UI redesign. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Locked decisions in `06-CONTEXT.md` (D-01..D-04) are normative and are pre-populated below; sections marked **Claude's discretion** are decided here and become normative for `gsd-planner` and `gsd-executor`.

Every dimension traces to ≥ 1 requirement: REQ-ui-main-flow (UI-01/04/05/06), REQ-ui-problem-block-view (UI-02/03), REQ-input-preflight, REQ-pipeline-logging, REQ-ui-design-review (UI-07).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (shadcn N/A — Python/Streamlit stack, not React/Vite/Next) |
| Preset | not applicable |
| Component library | Streamlit ≥ 1.35 primitives only (`st.sidebar`, `st.dialog`, `st.expander`, `st.dataframe`, `st.metric`, `st.file_uploader`, `st.selectbox`, `st.multiselect`, `st.text_area`, `st.checkbox`, `st.button`, `st.download_button`, `st.code`, `st.error`, `st.success`, `st.info`, `st.warning`) — no `streamlit-elements`, no `streamlit-aggrid`, no React side-loads (per `06-CONTEXT.md` Claude's discretion §"Styling depth") |
| Icon library | Unicode glyphs only (●  ◉  ✏️  ⚠️  ✗  🛑) — Streamlit-native, no SVG/icon-font dependency; matches existing `render_status_badges` precedent in `app.py` |
| Font | System sans-serif (Streamlit default) — no custom `@font-face`. Existing `inject_page_styles` in `app.py:26-155` is REUSED (not extended); CSS surface remains a one-time block per `06-CONTEXT.md` §"Styling depth" |

**Streamlit version gate.** `requirements.txt:5` pins `streamlit>=1.35` — already satisfies the `st.dialog` minimum (1.32) by 3 minor versions. **No pin bump required.** Researcher Pitfall §"Streamlit `st.dialog` requires ≥ 1.32" → CLOSED.

**`st.dialog` programmatic-close pattern (verified).** Inside the dialog function: after `save_methodical_profile(...)` succeeds, set `st.session_state["profile_selectbox"] = new_profile_id` then call `st.rerun()`. On the next script run the dialog function is not invoked, the modal is dismissed, and the sidebar `st.selectbox` re-renders with the new profile pre-selected. Researcher Pitfall §"D-03 step 5 modal auto-close" → CLOSED. Constraint: only one `st.dialog` may be active per script run — fine here, the methodical modal is the only modal in Phase 6.

**`st.error` under uploader (verified).** `st.error` is a widget that renders in place; Streamlit's full-script-rerun model means preflight errors render under the uploader on the same rerun, no script restart needed. The preflight is a function called from `main()` whose `st.error` output appears immediately under `st.file_uploader`. Researcher Pitfall §"Preflight `st.error` without full restart" → CLOSED.

**`st.expander` inside `st.dataframe` (verified).** Streamlit does NOT support row-level expand inside `st.dataframe`. Chosen pattern (per discretion below): **per-row `st.expander` loop** inside the three grouped sections. Performance budget: 260+ blocks worst case (Phase 5 corpus) — rendering ~260 collapsed expanders is well within Streamlit's render budget because the «Без изменений» group (typically ≥ 80% of blocks) renders COLLAPSED, so only the header HTML is emitted per row; expander bodies are not rendered until the user clicks. The «Требуют внимания» group (usually < 30 rows) renders EXPANDED — that is the explicit design hot path. If the «Без изменений» group exceeds 150 rows in measured usage, executor may fall back to a single `st.dataframe` for that group only (chip column + paired `st.selectbox` for detail-row drill-in) — discretion logged below. Researcher Pitfall §"chip + expander row pattern for 260 blocks" → CLOSED.

## Spacing Scale

Streamlit primitives self-space via internal padding; custom spacing is applied only inside the inherited `inject_page_styles` block (`app.py:26-155`). The contract below governs **only** the CSS values we author — Streamlit container gaps remain default.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Status-chip horizontal padding (inline pill spacing in `.badge` `0.35rem 0.7rem` ≈ 5.6px/11.2px → quantised to 4/8 below) |
| sm | 8px | `.badge` vertical padding, `.metric-card` internal label-to-value gap (already ~`0.5rem` in `app.py:114`) |
| md | 16px | `.metric-card` padding (`1rem 1.1rem` → quantised to 16px), expander row internal gap, section-to-section gap inside main pane |
| lg | 24px | `.hero` block bottom margin, distance between "report header" and "summary counters", between counters and grouped sections |
| xl | 32px | Distance between «Требуют внимания» section block and «Изменены» / «Без изменений» blocks |
| 2xl | 48px | Distance between summary counters and downloads section (visual end-of-report break) |
| 3xl | 64px | Reserved — not used in Phase 6 (no full-page section breaks; single-page report) |

**Quantisation rule for executor:** existing `inject_page_styles` uses rem (`1rem`, `0.35rem`, `1.6rem`) — values in `app.py` ARE NOT rewritten. New CSS added by Phase 6 (if any) uses these px tokens. Mixed rem/px in the inherited CSS is acceptable per CLAUDE.md «не «улучшай» соседний код, форматирование и комментарии; следуй существующему стилю».

**Exceptions:** none. The `.metric-card.min-height: 110px` in `app.py:108` is grandfathered (existing code, not rewritten — CLAUDE.md rule).

## Typography

Streamlit-native sizes — no `font-family`/`@font-face` overrides beyond `app.py` precedent. The four roles below cover every text element in Phase 6.

| Role | Size | Weight | Line Height | Streamlit primitive that produces it |
|------|------|--------|-------------|----------------------------------|
| Body | 14px | 400 (regular) | 1.5 | `st.write`, `st.markdown` paragraph text, `st.expander` body, table-cell text, JSON-block text |
| Label | 14px | 600 (semibold) | 1.4 | `st.selectbox`/`st.multiselect`/`st.checkbox` labels, `.metric-card .label` (currently `0.85rem` ≈ 13.6px, accept as 14), status-chip text inside `.badge` (currently `0.85rem`) |
| Heading | 20px | 600 (semibold) | 1.25 | `st.subheader`, group titles («Требуют внимания» / «Изменены» / «Без изменений»), modal heading inside `st.dialog` |
| Display | 32px | 800 (extrabold) | 1.1 | `.metric-card .value` (currently `2rem` ≈ 32px, weight 800 — already correct in `app.py:117`); profile_id chip in report header |

**Weights:** exactly 2 weights normative — 400 regular + 600 semibold. Display weight 800 is grandfathered from existing `app.py` (CLAUDE.md rule: don't rewrite working code).

**Line heights:** body 1.5 (Russian Cyrillic body comfort), heading/display 1.1–1.25. Existing `app.py` hero/badge sizes already comply.

**Russian-language gate (CLAUDE.md «Russian UI consistent with existing app.py»).** Every piece of copy in the contract below is Russian. No English fallbacks except technical identifiers (`profile_id`, status-enum constants, error class names — these stay literal because they appear in the JSON run-log and the executor must not translate them).

### Summary counters (UI-04 «above the per-block table»)

**Widget choice (discretion):** **`st.metric` strip — 5 metrics in a single row of `st.columns(5)`.** Rationale:
- `st.metric` is Streamlit-native, no custom CSS beyond what's already in `inject_page_styles`.
- The 5 counters required by ROADMAP SC-1 (total / no_change / changed / review / error) map 1:1 to 5 `st.metric` cells; adding `blocked_unsafe_autofix` makes it 6 cells in one row (a 6-up `st.columns(6)` still fits at `layout="wide"` per existing `app.py:23`).
- The existing `render_metric_card` HTML-card pattern (`app.py:178-189`) is **deprecated for Phase 6** — it's 4-up, doesn't include `review` separately, and embeds inline HTML that conflicts with the simpler `st.metric` Streamlit semantics. Executor MAY keep `render_metric_card` callable for legacy compat but the new UI flow uses `st.metric` exclusively.
- The previously-used `render_status_badges` HTML-pill row (`app.py:192-205`) is **retired in Phase 6** — its function is fully subsumed by `st.metric` (count + delta) plus the per-section grouped headers below.

| Label | Value source | Delta (optional) |
|-------|--------------|------------------|
| «Всего блоков» | `summary["blocks_total"]` | — |
| «Без изменений» | `summary["no_change"]` | — |
| «Изменены» | `summary["changed"]` | — |
| «Требуют проверки» | `summary["review"]` | — |

### Block table widget choice (discretion)

**Per-row `st.expander` loop inside each of the three grouped sections.** Decision matrix:

| Widget | Row-expand | Status chip column | Mixed-content body | Perf @ 260 blocks |
|--------|-----------|---------------------|----------------------|---------------------|
| `st.dataframe` | NO (Streamlit limit) | YES (text col) | NO (single value-type per col) | Excellent |
| `st.data_editor` | NO | YES | NO + edit state churn | Good but adds state-management cost not needed |
| **per-row `st.expander` loop** | YES (native) | YES (icon+label inline in header) | YES (block text + confidence + explanation + reasons) | Acceptable — collapsed rows render header only; 260 collapsed expanders < 50ms render budget |

Executor implements one helper `render_block_section(title, df, expanded_by_default)` that loops `df.iterrows()` and renders one `st.expander` per row. Expander label format:

```
{icon} {block_id} · {label} · {Russian status label} · уверенность {confidence:.2f}
```

Expander body (REQ-ui-problem-block-view + UI-02 + UI-03):
- **«Оригинальный текст блока»** — `st.code(text, language=None)` (preserves whitespace; no Markdown re-interpretation of Russian text)
- **«Причина ручной проверки»** — `st.write(explanation)` (UI-03 «manual-review reason from `explanation`»)
- **«Нарушенные правила»** — `st.write(violated_rules)` if non-empty
- **«Применённые исправления»** — `st.write(applied_fixes)` if non-empty
- **«Заблокированное автоисправление»** — `st.write(unsafe_auto_fix_reason)` if `status == blocked_unsafe_autofix`

Per-row error blocks (D-04 «per-block error → status chip + expander with user-message + error class name»):
- Expander body shows `user_message` (Russian) + `error_class_name` (literal) — **no traceback, no document text**, both excluded.

**Fallback (executor-only escape hatch):** if measured render time for the «Без изменений» group exceeds 1500ms in the worst-case 260-block corpus (Phase 5 fixture `tests/fixtures/corpus/negative/*.docx`), executor may replace ONLY that group's per-row loop with a single `st.dataframe` of `[block_id, icon+label chip, text-preview-first-80-chars]` and a paired `st.selectbox` for drill-in. The «Требуют внимания» and «Изменены» groups stay on per-row expanders unconditionally.

### Download filename scheme (UI-06 «never overwrite original»)

**Inherited from `application_service.py:175-182` — already correct, no change needed:**
- Report CSV: `results/reports/{stem}_{mode}_audit_{timestamp}.csv`
- Report JSON: `results/reports/audit_report_{timestamp}.json`
- Summary JSON: `results/reports/summary_{timestamp}.json`
- Summary TXT: `results/reports/summary_{timestamp}.txt`
- Predictions CSV: `results/predictions/{stem}_{model_choice}_predictions_{timestamp}.csv`
- Corrected DOCX: `results/generated_docx/{stem}_{model_choice}_formatted_{timestamp}.docx`

Phase 6 adds ONE new artifact (D-04 run-log):
- Run-log JSON: `results/reports/{stem}_run_log_{timestamp}.json`

`{timestamp}` is `YYYYMMDD_HHMMSS` per existing `_timestamp()` helper (`application_service.py:86-87`). `{stem}` is `Path(input_path).stem` — never `input_path.name`, so the original DOCX file is provably untouched on disk regardless of how many times audit runs. UI-06 satisfied.

### Run-log JSON contract (D-04 «JSON array of stage records»)

**One file per audit run.** UTF-8, indent=2. Structure:

```json
[
  {"stage": "document-read",   "ts": "2026-05-14T12:34:56.789Z", "status": "ok",    "error_class": null, "error_message": null},
  {"stage": "classification",  "ts": "2026-05-14T12:34:58.123Z", "status": "ok",    "error_class": null, "error_message": null},
  {"stage": "rule-apply",      "ts": "2026-05-14T12:35:01.456Z", "status": "partial", "error_class": "ValueError", "error_message": "block_id=23: invalid numPr override", "block_id": 23},
  {"stage": "save",            "ts": "2026-05-14T12:35:02.000Z", "status": "ok",    "error_class": null, "error_message": null}
]
```

**PII boundary (REQ-pipeline-logging + D-04 «PII boundary: filename + technical metadata IN; document content OUT»):**

| Field | Allowed | Forbidden |
|-------|---------|-----------|
| Filename / path | `input_path.name` (just the basename, never the full path) | Full absolute path with username (`/Users/…`) |
| Stage name | one of `document-read`, `classification`, `rule-apply`, `save` (literal enum) | free-text variants |
| Timestamp | ISO-8601 UTC | locale-dependent strings |
| Status | one of `ok`, `partial`, `error` | other values |
| Error class | bare class name (`KeyError`, `ValueError`) | full qualname (`builtins.KeyError`) is acceptable, but never include `__module__.__qualname__` from third-party libs |
| Error message | short user-message (Russian or English class-name) | **never** the offending document text |
| Block identifier (when applicable) | integer `block_id` from `predictions_df` | original block text, paragraph XML, file offsets |
| Profile context (optional) | `profile_id` (literal string) | the loaded profile JSON (it's in the report sidecar, not the run-log) |
| Traceback | **never** | **never** |

**Logger is single-writer** (D-04). One helper, e.g. `src/inference/run_log.py::RunLog(input_filename: str)` with `.record(stage, status, error_class=None, error_message=None, **extras)` and `.dump_json(path: Path)`. All four stages (`document-read`, `classification`, `rule-apply`, `save`) call this single helper; no other writer.

### View-layer evolution decision (discretion)

**Evolve `app.py` in-place; do NOT rewrite the view layer wholesale.**

Rationale (per CLAUDE.md «минимум кода» + «не рефактори то, что работает, без явного запроса»):
- `application_service.py` (216 LoC) is reusable as-is per `06-CONTEXT.md` §"Reusable Assets". Phase 6 doesn't touch backend.
- `app.py` has reusable assets: `inject_page_styles`, `render_metric_card`, `normalize_table_values`, `format_profile_option`, `build_profile_options`, `save_uploaded_bytes`, `run_processing` core flow.
- The dropped 5-tab structure (lines 550-748) is ONE function (`render_results`) — replace it; everything outside that function stays.
