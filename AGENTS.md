# GOST Document Formatting Project

## Self-improvement protocol
- После любой моей коррекции ты обязан предложить лаконичное правило и дописать его в этот CLAUDE.md в подходящую секцию.
- Формат правила: одно императивное предложение, без обоснования и без примеров, если случай не двусмысленный.
- Перед добавлением правила выполни поиск по этому файлу. Если уже есть правило, покрывающее этот случай, не дублируй, а уточни существующее.
- Держи общий объём CLAUDE.md в пределах 2500 токенов. При превышении объедини пересекающиеся правила и удали устаревшие.
- После фикса бага записывай правило про класс корневой причины, а не про конкретный симптом.

## Принципы исполнения

- Не делай поспешных выводов; формулируй допущения вслух и спрашивай, если не уверен.
- Если интерпретаций несколько — предложи все, не выбирай молча.
- Используй явно указанный пользователем источник данных и не подменяй его более новым или удобным каталогом.
- Перед safe-formatting сверяй labels модели с извлечённой DOCX-структурой.
- Для автонумерации DOCX применяй Word numbering (`numPr`), а не только стиль абзаца.
- Для многораздельной библиографии передавай номер раздела через postprocess metadata до rule engine.
- В regression-аудите применяй тот же postprocess структурных labels, что и в основном inference-пайплайне.
- Преобразуй таб-ведомые короткие серии после вводной фразы с двоеточием в `list_item` до safe-formatting.
- Если есть более простой подход — скажи; отказывайся от него только при обосновании.
- Пиши минимум кода, необходимого для задачи: никаких лишних абстракций, «гибкости», обработки невозможных сценариев.
- Если можно решить в 50 строк вместо 200 — переписывай.
- Не «улучшай» соседний код, форматирование и комментарии; следуй существующему стилю.
- Не рефактори то, что работает, без явного запроса.
- Не перезаписывай наследуемое DOCX-форматирование прямыми значениями без regression-теста на сохранение Word-стилей.
- Восстанавливай отсутствующий или сломанный `numPr` у списка, не меняя уже принятую раскладку абзаца.
- Для `body_text` с accepted list layout восстанавливай `numPr`, если абзац короткий и без явного номера.
- Не удаляй ведущие табы из текста до постпроцессинга, если они нужны для распознавания списка.
- Не обрезай сырой `text` из predictions CSV до передачи его в rule engine.
- Удаляй orphans, появившиеся из-за твоих изменений; чужой мёртвый код не трогай — упомяни.
- Каждая изменённая строка должна напрямую отслеживаться до запроса пользователя.
- Превращай задачи в проверяемые цели; для многоэтапных задач выдавай план «шаг → verify».

## Тестирование (TDD + дизайн под тесты)

**Железный закон:** никакого продакшен-кода без падающего теста. Код, написанный раньше теста, — удалить и начать заново.

**Цикл RED-GREEN-REFACTOR:** RED — один падающий тест, падает по нужной причине; GREEN — минимум кода, ВСЕ тесты зелёные; REFACTOR — чисти, прогоняй после каждого шага. Повтори.

**Дизайн под тесты:** чистую логику отделяй от I/O; внешние зависимости через DI; каждая фича доступна через CLI/API, не только GUI; маленькие функции на одну ответственность («and» в имени теста = разделить); возвращай значения, не печатай; тесты запускаются независимо, в любом порядке, параллельно и убирают за собой.

**Качество и порядок:** имя описывает поведение (`rejects_empty_email`, не `test1`); мокай только неизбежный внешний I/O, предпочитай реальные зависимости; порядок — контрактные → интеграционные → юнит; покрытие — каждый публичный эндпоинт (контракт), каждый workflow (happy + 2 error path), каждое ветвление (юнит на ветку), крайние случаи (null/пусто/границы/конкуренция) отдельно.

**Багфикс:** начинается с падающего теста, воспроизводящего баг.

**Верификация «готово»:** никогда не утверждай «тесты проходят» без свежего запуска и чтения вывода в текущей сессии. Чек: каждая функция имеет тест; тест наблюдался падающим по ожидаемой причине; написан минимум кода; все тесты зелёные сейчас; вывод чистый (без warnings/errors).

**Стоп-сигналы:** «слишком просто для теста», «напишу тесты потом», «TDD замедлит», «только в этот раз», «уже проверил вручную», «оставлю как референс».

## Code Review (только diff текущего изменения)

Проходи линзы по порядку; останавливайся на CRITICAL.

**Корректность:** все критерии приёмки реализованы и не заглушки; нет расширения скоупа без обоснования; обработаны happy/unhappy/error path и крайние случаи.

**Безопасность (OWASP в рамках diff):** новые маршруты под auth + проверка владения (нет IDOR); все запросы к БД параметризованы, нет `eval`/`exec`/shell injection; нет захардкоженных секретов; нет PII/учёток в логах и ошибках; токены сессии перегенерируются после логина; cookies `HttpOnly`+`Secure`+`SameSite`; внешний ввод валидируется на сервере.

**Архитектура:** структура директорий, стек и границы модулей соблюдены; нет нарушения слоёв (UI не лезет в БД); обработка ошибок консистентна; нет N+1; все I/O закрываются на каждом пути.

**Поддерживаемость (только реальные проблемы):** функции > ~60 строк с очевидной точкой разделения; дублирование логики; коммит мёртвого или закомментированного кода.

**Серьёзность:** `CRITICAL` (security/data-loss/нарушение требования) → блокирует merge; `HIGH` (значимый баг или арх. нарушение) → блокирует merge; `MEDIUM/LOW` → не блокирует.
