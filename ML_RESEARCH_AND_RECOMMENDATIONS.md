# ML-исследование и рекомендации для gost_formatter

## Вывод

Рекомендуемая ML-часть системы: **Linear SVM + TF-IDF + форматные признаки + обязательный rule-based слой ГОСТ**.

## Почему не RuBERT как основное ядро

RuBERT хорошо работает, когда класс определяется семантикой текста. В задаче распознавания блоков документа класс часто определяется не только текстом, но и оформлением: стилем, выравниванием, жирностью, длиной строки, нумерацией и типом блока. На небольшом корпусе RuBERT требует больше данных и ресурсов, а также хуже использует форматные признаки DOCX без дополнительной архитектуры.

## Сравнение вариантов

| Вариант | Плюсы | Минусы | Роль в системе |
|---|---|---|---|
| Rule-based only | объяснимо, быстро | плохо восстанавливает неявные роли | только ГОСТ-слой |
| Logistic Regression + TF-IDF | простой baseline | часто слабее SVM | контрольный эксперимент |
| SVM + TF-IDF | сильный baseline | не использует оформление | эксперимент |
| SVM + TF-IDF + formatting | лучший баланс | зависит от качества признаков | основной вариант |
| RuBERT | учитывает контекст | ресурсоемкий, требует больше данных | исследовательская альтернатива |

## Рекомендуемый pipeline

```text
DOCX / PDF с текстовым слоем
  -> извлечение блоков
  -> нормализация текста
  -> TF-IDF + форматные признаки
  -> Linear SVM
  -> postprocessing-эвристики
  -> rule-based ГОСТ-проверка
  -> отчет / ограниченная коррекция DOCX
```

## Граница ответственности

ML отвечает за функциональную роль блока. Rule-based слой отвечает за нормативную проверку и исправление.
